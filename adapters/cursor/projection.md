## Strategy

**Rules-only first.** Project canonical skill content into MDC rule files:

- Iron-law rules → `.cursor/rules/<name>.mdc` with `alwaysApply: true`.
- Skill bodies → `.cursor/rules/<skill-name>.mdc` with `globs: <relevant-paths>` or `description: <when-to-load>` so Cursor selects them by context.
- Language overlays → `.cursor/rules/lang-<lang>.mdc` with `globs: '**/*.{ts,tsx}'` etc.

Hooks projected separately (best-effort) under a Cursor-specific manifest with explicit "unverified shape" disclosure.

## MDC Frontmatter

Cursor MDC files use frontmatter for scoping:

```yaml
---
description: <when to load this rule>
globs: <comma-or-array of glob patterns>
alwaysApply: true | false
---
```

## Generation

Generated by `scripts/project.mjs` from canonical. Generated MDC files ARE committed to the repo so Cursor reads them without running the script. Source-of-truth remains canonical; MDC is downstream.

## Statuslines

Cursor's built-in IDE status row (model badge, token bar) is not user-scriptable — there is no extension point to render an Aegis statusline preset into it. Aegis statusline presets are canonical (`statuslines/`) and first-class on Claude Code; on Cursor this is an honest gap.

## Permissions

No projection. Cursor has **no host primitive for agent-level tool restriction** —
canonical agents project to MDC rules (no native subagent primitive), and there is
no per-agent `tools` allowlist or permission block. Aegis agent permissions are
therefore **advisory** on Cursor: `manifest/permissions.json` documents the
intended posture, but nothing enforces it at runtime here. Aegis does not write
enforcement prose into a rule that the host won't honour. (Decision D6.)

- **Source of truth:** `manifest/permissions.json`.
- **Author guide:** `docs/agent-permissions.md`.
- **Enforced hosts:** Claude (`tools:` allowlist) and OpenCode
  (`agent.<name>.permission`) — see those `adapters/<host>/projection.md`.

## Dynamic workflows (gap)

Claude Code's dynamic workflows are a host-resident built-in (`Workflow` tool): the model writes a JS orchestration script and a background runtime fans it out across dozens-to-hundreds of subagents, saving reusable scripts to `.claude/workflows/`. This is not a plugin extension point — Aegis cannot ship, declare, or project it, and no equivalent exists on this host. The portable substitute is the `orchestrate` skill (≤5-wave `Task()` fan-out with in-session synthesis), which runs identically everywhere but does not reach the hundreds-of-agents, context-isolated, resumable regime. For very large audits/migrations on this host, the ≤5-wave skill is the ceiling.
