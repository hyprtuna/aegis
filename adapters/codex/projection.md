**Status:** shipped. Plugin layout modernized in a later release. Hooks are a declared honest gap — see "Hook capability matrix" below.

Generator: `node scripts/project.mjs` → `projectCodex()` + `projectCodexPluginManifest()` + `projectCodexMarketplaces()`. There is no hook generator — see "Deleted: the Codex hook projector" below. Install doc: `.codex/INSTALL.md`. Plugin root: `.codex/plugins/aegis/`. Plugin manifest: `.codex/plugins/aegis/.codex-plugin/plugin.json`. Rules surface: `.codex-plugin/AGENTS.md` (project root — unchanged). MCP stub: `.codex/plugins/aegis/.mcp.json`. No hook bundle ships today — no canonical `hooks/*.json` intent binds `codex` (see Hook capability matrix).

## What Lands on Codex

- **`.codex/plugins/aegis/skills/aegis-<name>/SKILL.md`** — one Codex Agent Skill folder per canonical skill, agent (folded), and command (as dispatcher). 79 canonical skills + 17 agents-as-skills + 6 command-as-dispatcher skills, all `aegis-` prefixed.
- **`.codex/plugins/aegis/.codex-plugin/plugin.json`** — official plugin manifest at the plugin root (D-04). Generated by `projectCodexPluginManifest()`; fields: `name`, `description`, `version`, `homepage`, `license`, `author`, `keywords` (renamed from `tags`), `skills:"./skills/"`, `mcpServers:"./.mcp.json"`. `version` synced from `package.json`.
- **`.codex/plugins/aegis/.mcp.json`** — MCP server stub at plugin root (D-06). Empty `{"mcpServers":{}}` — real servers deferred per MCP policy.
- **`.agents/plugins/marketplace.json`** — Codex marketplace manifest, and the **sole** Codex discovery path. Uses object source form `{source:"local", path:"./.codex/plugins/aegis"}` required by `codex plugin add`. Codex no longer shares `.claude-plugin/marketplace.json`: that file is now Claude's alone (string `source: "./"`) — writing the Codex object shape there broke `claude plugin validate`, so the two hosts get separate marketplaces with no cross-host leak.
- **`.codex-plugin/AGENTS.md`** — concatenated rules surface. Iron Laws block at top, then `# Rules` H1, then every canonical `rules/*.md` body under an `## <rule-name>` H2. Read natively by Codex at project root.

### Install-verified layout (D-04, live codex 0.141.0, 2026-06-20)

`codex plugin marketplace add <path>` → `codex plugin add aegis@<mp>` → status **`installed, enabled`**. Skills preserved at `~/.codex/plugins/cache/$MP/$PLUGIN/$VERSION/skills/<name>/SKILL.md`. The marketplace manifest MUST use the object `source` form; the string `"./"` form errors "plugin was not found in marketplace". Verified against a minimal official-layout plugin (`.codex-plugin/plugin.json` with `skills:"./skills/"` + `skills/<name>/SKILL.md`).

## Locked Decisions

- **`aegis-` prefix on every projected skill name** to dodge collisions with Codex reserved names (`default`, `worker`, `explorer`) and across plugins. Reserved-name guard hard-fails at projection time.
- **Agent-as-skill folding** — `agents/<name>.md` projects to `.codex/plugins/aegis/skills/aegis-<name>/SKILL.md`. Codex *does* have a native subagent primitive (`.codex/agents/*.toml`), but it is **not plugin-distributable** (D-02, verified 2026-06-20), so a plugin-shipped Aegis cannot deliver it — folding is the viable path. Agents become discoverable skills users invoke explicitly (runtime-verified: the install test's skill list included agent-derived skills). They lose dispatchable-subagent semantics on Codex.
- **Command-as-dispatcher pattern** — `commands/<name>.md` projects to a thin `SKILL.md` (~10–15 line body) referencing the canonical skill/workflow path. Skill+command name collisions resolve with a `__command` suffix on the command-derived skill.
- **Plugin root at `.codex/plugins/aegis/`** — self-contained plugin directory; manifest at `.codex-plugin/plugin.json` inside it, skills at `skills/`, MCP stub at `.mcp.json`. This is the official Codex plugin layout verified against codex 0.141.0 (D-04).
- **Object source form in marketplace** — `{source:"local", path:"./.codex/plugins/aegis"}` required for `codex plugin add` to resolve the plugin. The string `"./"` form is broken (D-05).

## Removed Decisions

- **8 KB skill-body cap** — removed once it was found to be an unsourced guardrail targeting the wrong unit. The real Codex budget is ~8,000 characters on the aggregated skills *list* (names+descriptions), not per-body — the full SKILL.md body is read with no truncation once selected.

## Persistent Memory (gap)

Codex has no native subagent `memory` primitive equivalent to Claude's `memory:` frontmatter field.
`x-claude.memory` is NOT emitted by the Codex projector.

**Fallback:** use a plain `.aegis-memory/MEMORY.md` file at the project root as the cross-host
memory store. The `recall` skill's Layer 1 (Grep) and Layer 3 (Read) work on Codex — only
Layer 2 (Claude's 200-line auto-injection) is absent. Start with Layer 1 on this host.

**Decay:** real decay / last-referenced scoring needs a store — deferred. Honest gap.

## Honest Gaps

- **Real MCP servers** — only an empty `mcp.json` stub ships today. First Aegis-supplied servers land in a later release.
- **Persistent memory** — no native `memory:` primitive; fallback is `.aegis-memory/MEMORY.md` read by the `recall` skill. See Persistent Memory section above.
- **Hook lifecycle (feature-removed gap)** — no canonical hook binds `codex` today. Codex's `[features] plugin_hooks` is **removed** (not merely disabled) — verified live against `codex features list` on codex-cli 0.144.6, which reports `plugin_hooks   removed   false`. A plugin cannot ship a hook that fires on this host, in any context: `hooks: stable` in that same feature list is Codex's *user*-configured `config.toml` hooks, a channel plugins have no access to. Aegis previously projected `session-start`/`pre-compact`/`post-compact` to Codex `hooks/hooks.json`; `codex` was dropped from all three intents' `platforms` and their `x-codex` bindings, because a binding that cannot fire is worse than a declared gap (Iron Law 6). As of v0.2.1 the projector path itself is deleted and `codex` is **rejected** outright in a hook intent's `platforms` — see the Hook capability matrix below. `instructions-loaded` was already a gap for the unrelated reason that Codex has no `InstructionsLoaded` event counterpart. Aegis's Codex bootstrap is unaffected — it continues through native Skill discovery (`.codex/plugins/aegis/skills/`) and the `.codex-plugin/AGENTS.md` project-root read, neither of which depends on plugin hooks. Revisit if a future Codex release un-removes `plugin_hooks`.
- **Statusline** — no native Codex statusline slot. Aegis ships statusline presets (Claude Code first-class); Codex stays gap-documented.
- **Distinct subagent semantics** — Codex *has* a native subagent primitive (`.codex/agents/*.toml`, project/personal-scoped), but plugins **cannot bundle agents** (the manifest exposes only `skills`/`hooks`/`mcpServers`/`apps`; no `agents/` in the plugin layout — verified against `developers.openai.com/codex/plugins/build`, 2026-06-20). A plugin-distributed Aegis therefore can't ship native subagents; they stay folded into skills. A user wanting native Aegis subagents must hand-place TOMLs under their own `.codex/agents/` (out of scope). Revisit if Codex adds an `agents` plugin-manifest pointer.
- **HTML/JSON template runtime reachability — `partial`.** `project.mjs` bundles `templates/html`, `templates/json`, and `manifest/template-index.json` into `.codex/plugins/aegis/` (fixing the prior gap where `:json` bundled-pointers, e.g. `design-system`/`plan-audit-report`, referenced files that never shipped on Codex). The build-time `${TEMPLATE:*}` directive still resolves correctly at projection (it emits a bundled-path pointer, per `scripts/project.mjs:231-257`). But the new runtime index→Read procedure (`rules/user-choice-discipline.md` step 5) reads `${CLAUDE_PLUGIN_ROOT}/...` to locate the chosen template, and Codex has **no equivalent base-path token** (`scripts/project.mjs:234-241` never emits `${CLAUDE_PLUGIN_ROOT}` for `host === "codex"`). An agent following step 5 on Codex cannot resolve the runtime read, so HTML/JSON selection stays `partial` — same limitation class as OpenCode (see `adapters/opencode/projection.md`'s templates-gap section). Only Claude closes the runtime read end-to-end.
- **`@rules/<file>.md` agent hotlinks** — three skeptical agents (`code-reviewer`, `code-quality-reviewer`, `doc-verifier`) reference `@rules/skeptical-stance.md` so Claude auto-inherits the shared rule text. Codex does **not** resolve `@`-includes; the literal `@rules/...` ships as plain prose. Mitigation: each agent retains a one-line inline summary of the stance, so the doctrine is not lost — only the auto-inherited full text is. Honest gap, not a silent drop.
- **config-schema.json** — No published machine JSON schema for Codex's `config.toml` is referenced in the official docs; the config-reference page at `developers.openai.com/codex/config-reference` is the schema. Aegis vendors the config-reference doc (see `references/codex-docs/`) rather than a machine schema file. If a `config-schema.json` ships in the `openai/codex` GitHub repo in a future release, update this gap and vendor it then.

## Hook capability matrix

Codex's `[features] plugin_hooks` flag is **removed**, not merely off-by-default —
verified live against `codex features list` on **codex-cli 0.144.6**:

```
hooks                stable     true
plugin_hooks         removed    false
```

`hooks: stable` is Codex's *user*-configured hook mechanism (`config.toml`, set by
the person running Codex) — a real, working feature. `plugin_hooks` is the distinct
flag that would let a *plugin* ship hooks that fire, and it does not exist as a
runtime path. This is stronger than "unverified in CI" — **no plugin-shipped hook
fires on Codex, in any context**, so none of Aegis's lifecycle hook intents bind
`codex` in `platforms` (all were dropped `x-codex` bindings that previously existed).

Note the precise claim. Registration and execution are different things: Codex's
`load_plugin_hooks` *does* discover and register a plugin's `hooks/hooks.json` when
the manifest declares no `hooks` field, and that registration can surface an
install-time trust prompt. What `plugin_hooks: removed` gates is **execution**. So
the defensible statement is that no plugin-shipped hook *fires* — not that a bundled
`hooks.json` is inert in every sense. This strengthens the decision below rather
than weakening it: a hook bundle could have cost users a trust prompt while being
incapable of ever running.

Prior art (verified against the vendored `references/superpowers/` 6.1.1 tree):
Superpowers added native Codex hooks, removed them 14 days later, and now ships
`"hooks": {}` in `.codex-plugin/plugin.json:24`. They have **no Codex hook projector
anywhere** in the tree. Their `hooks/` directory still reaches the Codex plugin as a
*packaging artifact* (`scripts/sync-to-codex-plugin.sh:407` rsyncs it; `EXCLUDES` at
`:45-65` does not filter it), but their packaged Codex archive excludes `^hooks/`
outright (`tests/codex/test-package-codex-plugin.sh:166`) and their own
`hooks/hooks.json:9` is Claude-shaped (`${CLAUDE_PLUGIN_ROOT}`), inoperable on Codex.
The direction of that evidence is unambiguous: they do not project hooks to Codex,
and where hook files land there they work to *neutralise* them. Aegis's
`projectCodexPluginManifest()` emits the same `{}` suppression (see Surfaces &
Verification above).

### Deleted: the Codex hook projector (v0.2.1)

Aegis previously carried a complete Codex hook path that nothing could exercise —
no intent bound `codex`, so it was dead code by construction. Across three review
rounds it produced four defects, none findable by a test. It was **removed**, and
`codex` became a *rejected* value in a hook intent's `platforms` rather than an
accepted-but-inert one — a value that validates and projects nothing is a silent
failure, strictly worse than the dead code it replaced.

**Re-add cost, should `plugin_hooks` ever return.** Recoverable from git history;
this is what would have to come back, and all of it must move as one unit:

| Piece | Where it lived |
|---|---|
| `projectCodexHooks()` — hooks.json emitter + script bundler + prune | `scripts/project.mjs` |
| `filterCodexHookIntents()` | `scripts/project.mjs` |
| `x-codex` hand-validation inside `loadHookIntents()` | `scripts/project.mjs` |
| `CODEX_EVENTS` + the `platforms.includes("codex")` validator branch | `scripts/validate/hook-intent.mjs` |
| `xCodex` `$def`, the `x-codex` property, the codex `allOf` clause | `manifest/schemas/hook-intent.schema.json` |
| FIX-V9 bundled-script byte-equality drift gate | `scripts/validate/codex.mjs` |
| The `hooks: "./hooks/hooks.json"` manifest pointer branch | `scripts/project.mjs` |

The bundler, the prune, and the drift gate are **three consumers of one directory**.
Every past defect came from moving one or two and leaving the third — so re-add all
three together or none.

**Trigger condition:** `codex features list` reporting `plugin_hooks` as anything
other than `removed`. Absent that, re-implementing from this record is cheaper than
maintaining a path nobody can run.

Aegis's Codex bootstrap is unaffected: native Skill discovery
(`.codex/plugins/aegis/skills/`) and the project-root `.codex-plugin/AGENTS.md` read
carry it without any hook.

## Statuslines

No documented Codex statusline. The bottom row of the Codex TUI is reserved for keybind hints and is not a user-scriptable status-row slot. Aegis ships statusline presets (Claude Code first-class); Codex stays an honest gap until Codex grows a documented statusline surface.

## Dynamic workflows (gap)

Claude Code's dynamic workflows are a host-resident built-in (`Workflow` tool): the model writes a JS orchestration script and a background runtime fans it out across dozens-to-hundreds of subagents, saving reusable scripts to `.claude/workflows/`. This is not a plugin extension point — Aegis cannot ship, declare, or project it, and no equivalent exists on this host. The portable substitute is the `orchestration` skill (≤5-wave `Task()` fan-out with in-session synthesis), which runs identically everywhere but does not reach the hundreds-of-agents, context-isolated, resumable regime. For very large audits/migrations on this host, the ≤5-wave skill is the ceiling.

## Permissions

No projection. Codex has **no host primitive for agent-level tool restriction** —
agents are folded into the skills tree (no distinct subagent primitive), and there
is no per-agent `tools` allowlist or permission block to emit into. Aegis agent
permissions are therefore **advisory** on Codex: `manifest/permissions.json`
documents the intended posture, but nothing enforces it at runtime here. Per
silent-failure discipline, Aegis does not write enforcement prose into
`.codex-plugin/AGENTS.md` that the host won't honour. (Decision D6.)

- **Source of truth:** `manifest/permissions.json` (the buckets, per-agent grants,
  and the plugin-level cross-cutting deny).
- **Author guide:** `docs/agent-permissions.md`.
- **Enforced hosts:** Claude (`tools:` allowlist) and OpenCode
  (`agent.<name>.permission`) — see those `adapters/<host>/projection.md`.

## Claude-only uptakes (gaps)

Several Claude-first capabilities have no Codex
counterpart. These are honest gaps; `manifest/capabilities.json` is the
authoritative per-capability host-status matrix.

- **`userConfig` install prompts** — Codex has no plugin-enable-time config-prompt
  mechanism; `${USER_CONFIG_*}` substitution is Claude-only.
- **Background monitors** — no Codex equivalent of plugin-declared monitor commands
  surfacing stdout as session notifications.
- **Plugin dependencies / semver constraints** — no Codex plugin-dependency manifest.
- **`.skill` (ZIP) distribution** — Codex install mirrors the git-spec pattern;
  the `dist/aegis.skill` ZIP channel is Claude-only.
- **`x-claude.skills` subagent preload** — Claude subagents support a `skills:`
  frontmatter list that preloads named skill bodies at subagent startup. Codex has
  no equivalent primitive; agents are folded into the skills tree (no distinct
  subagent frontmatter to carry the field). Claude-only; NOT emitted by the Codex
  projector.
- **Per-agent model (authoring-time, `manifest/permissions.json` D3)** — Claude agent
  frontmatter carries a `model:` field projected from `manifest/permissions.json`
  (D3 single-source; `x-claude.model` on canonical frontmatter is intentionally FORBIDDEN
  and the `project.mjs` `flattenXClaude()` hard-fail guard prevents re-introduction).
  Codex folds agents into skills and has no per-skill model-override field in its
  SKILL.md frontmatter. Claude-only; NOT emitted by the Codex projector.
- **Native subagent execution-profile fields** (`effort`, `maxTurns`, `background`,
  `isolation`) — Claude-plugin-only (cc-docs sub-agents.md, plugins-reference.md).
  Codex folds agents into skills with no analogous frontmatter; not emitted by the
  Codex projector.

See `manifest/capabilities.json` for the full matrix and per-host evidence.

## Generator

`scripts/project.mjs` → `projectCodex()`. Sub-functions:

- `projectCodexSkills()` — walks `skills/core/`, `skills/languages/`, `skills/workflows/`.
- `projectCodexAgentsAsSkills()` — walks `agents/*.md`.
- `projectCodexCommandsAsDispatchers()` — walks `commands/*.md`; applies `__command` suffix on collision.
- `projectCodexRules()` — concatenates `rules/*.md` into `.codex-plugin/AGENTS.md`.

Helpers: `prefixAegis(name)` for collision-avoidance prefixing, `extractIronLawsBlock()` for the rules surface, plus the `atomicReplace()` idempotency primitive. Reserved-name and name-shape guards are enforced inline via the `CODEX_RESERVED_NAMES` constant and `CODEX_NAME_PATTERN` regex (no dedicated helper function). Dispatcher bodies are composed inline inside `projectCodexCommandsAsDispatchers()`. Idempotent: re-running produces identical output.

- **No hook generator.** There is no `projectCodexHooks()` and no `.codex/plugins/aegis/hooks/` tree. `codex` is a rejected value in a hook intent's `platforms` — both `HOOK_INTENT` and the projector's `loadHookIntents()` hard-fail on it, so a hook can never silently target this host. See "Deleted: the Codex hook projector" above for what was removed and what re-adding would cost.
