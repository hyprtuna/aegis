# Structured Slate Extras — Frontmatter + Executable Plan Grammar

> Loaded by the `implementation-planner` skill when the user picks **Structured slate** or **Both** as the
> plan format (Q2), regardless of which location was chosen in Q1. Extends the generic implementation-planner
> body with structured slate format requirements, `<decisions>` D-NN grammar, plan-verifier
> compliance rules, and the `executable_plan` YAML schema.

## When This Addendum Applies

The user chose **Structured slate** or **Both** as the format (Q2). This triggers the following
regardless of the chosen location:

1. If location is `.aegis/plans/`: bootstrap with `mkdir -p .aegis/plans/` (silent, no confirmation needed).
2. Apply the structured slate format defined below.
3. `plan-verifier` compliance rules apply.
4. Plan validation and execution tooling integration is active.

## Mandatory `<plan-header>` Block

Every structured slate MUST include the full YAML frontmatter shape. Use this schema:

```yaml
<plan-header>
version: "<semver>"
title: "<Feature Name>"
status: "draft"
related_spec: "<path to spec file, if produced by brainstorm-spec>"
created: "<ISO-date>"
must_haves:
  truths:          # facts that must be true before this plan can execute
    - "<truth statement>"
  artifacts:       # files that must exist at plan completion
    - "<file path>"
  key_links:       # URLs or paths readers need for context
    - "<url or path>"
  covered_decisions:  # D-NN: IDs from the spec's <decisions> block
    - "D-01:"
    - "D-02:"
global_constraints:        # spec's project-wide requirements, exact values verbatim
  - "<one line each — version floors, naming/copy rules, platform reqs>"
</plan-header>
```

`covered_decisions` MUST list every `D-NN:` ID from the linked spec's `<decisions>` block.
`plan-verifier` will fail the audit if any `D-NN:` ID is absent from this list.

## `<decisions>` Block (Plan 30 Contract)

If this plan was produced from a spec generated by `brainstorm-spec`, the spec contains a
`<decisions>` block. Plans that have corresponding specs MUST include a reference to that
spec in `related_spec` so decision-coverage tooling can verify decision coverage.

This command checks that every decision in the `<decisions>` block is traceable to a task in
the plan. Never strip or rename the `<decisions>` block — it is a shipped contract.

### D-NN Decision ID Format

Decisions are identified using the format `D-NN` where `NN` is a zero-padded two-digit number
starting at `D-01`. When referencing a spec decision in a plan task, use the exact `D-NN:` ID
as it appears in the spec's `<decisions>` block.

## executable_plan YAML Schema

Structured slates MUST include an `executable_plan` YAML block immediately after the `<plan-header>`:

```yaml
executable_plan:
  tasks:
    - id: "T-01"
      title: "<Task Title>"
      phase: "<Phase Name>"
      files:
        - path: "src/core/config.ts"
          operation: "NEW"   # NEW | EXTEND | DELETE
      depends_on: []         # list of T-NN IDs this task depends on
      interfaces:
        consumes: ["<exact signatures this task uses from earlier tasks>"]
        produces: ["<exact names/types later tasks rely on>"]
      verification: "npm run typecheck && npm test -- --grep 'ConfigSchema'"
      acceptance: "typecheck passes, 3 new tests pass, config.ts exports ConfigSchema"
```

The `depends_on` field enables `plan-run` to determine execution order. Tasks with no
dependencies run first; tasks with dependencies run after their prerequisites complete.

## Composition Table

Structured slates MUST include a composition table near the top:

```markdown
## Composition

| Category | Count | Items |
|---|---|---|
| Technical debt | N | <items> |
| Improvements | N | <items> |
| Additions | N | <items> |
| Bug fixes | N | <items> |
| Docs/DX | N | <items> |
```

At least one category must have count > 0. Categories with count 0 MUST state why inline
(e.g., "no open bugs at cut", "no debt promoted this cycle").

## plan-verifier Compliance — Regex Blacklist

`plan-verifier` runs these checks on every structured slate. Failures block the plan audit:

| Check | Pattern | Why |
|---|---|---|
| No TBD | `/TBD/i` | Not a plan — it is a defer |
| No tbd | `/tbd/` | Same (case-insensitive) |
| No placeholder | `/<placeholder>/i` | Template stub never filled |
| No ellipsis | `/…/` | Implies omitted steps |
| No XXX | `/XXX/` | Needs fixing marker |
| No ??? | `/\?\?\?/` | Unresolved ambiguity |
| No lorem ipsum | `/lorem ipsum/i` | Filler text |
| No TODO | `/TODO/` | Reminder, not instruction |
| No implement later | `/implement later/i` | Scope deferral |
| No fill in later | `/fill in later/i` | Same |
| covered_decisions complete | all D-NN: IDs present | Decision traceability |
| related_spec present | `related_spec:` field | Links plan to spec |

## Structured Slate Full Format

```markdown
<plan-header>
version: "vX.Y.Z"
title: "Feature Name"
status: "draft"
related_spec: ".aegis/specs/features/<slug>/spec.md"
created: "YYYY-MM-DD"
must_haves:
  truths:
    - "spec approved by user"
  artifacts:
    - "src/feature/index.ts"
  key_links:
    - ".aegis/specs/features/<slug>/spec.md"
  covered_decisions:
    - "D-01:"
    - "D-02:"
global_constraints:
  - "Node >= 20; no Bun runtime"
  - "Named exports only; no default exports"
</plan-header>

executable_plan:
  tasks:
    - id: "T-01"
      title: "..."
      phase: "Phase 1"
      files:
        - path: "src/feature/index.ts"
          operation: "NEW"
      depends_on: []
      interfaces:
        consumes: []
        produces: ["Feature: (opts: FeatureOptions) => FeatureResult"]
      verification: "npm test -- --grep 'feature'"
      acceptance: "tests pass, index.ts exports Feature"

# Feature Name Implementation Plan

## Composition

| Category | Count | Items |
|---|---|---|
| Additions | 1 | New feature module |
| Technical debt | 0 | No debt promoted this cycle |
| Improvements | 0 | No improvements in scope |
| Bug fixes | 0 | No open bugs at cut |
| Docs/DX | 0 | Not in scope |

## Phase 1: <Name>

**Goal:** <one sentence>

### Task T-01 — <Task Title>

- **Files:** `src/feature/index.ts` [NEW]
- **Action:** <complete, concrete instructions>
- **Verification:** `npm test -- --grep 'feature'`
- **Acceptance:** tests pass, index.ts exports Feature
```

## Inline Self-Review (Structured-slate additions)

In addition to the base self-review checklist, for structured slates also verify:

- [ ] `must_haves.covered_decisions` complete? Every `D-NN:` ID from the spec's `<decisions>` block appears in the plan frontmatter.
- [ ] `executable_plan` tasks match narrative tasks? Every `T-NN` in the YAML has a corresponding `### Task T-NN` section in the plan body.
- [ ] `related_spec` field points to the correct spec file?
- [ ] Plan validation passes? Run plan-validate and confirm zero findings.
- [ ] Slash-parity preserved? If the plan adds or removes a slash command, the corresponding CLI command is added or removed in the same plan.
- [ ] Layered import rule honored? No plan task causes a lower-layer module to import from a higher-layer module.
- [ ] Global Constraints present (if the spec has project-wide reqs) and every task has an Interfaces block (consumes/produces with exact signatures)?

## Pass-through When Called by Another Skill

When `implementation-planner` is invoked by another skill (e.g., `default-feature`) with a pre-selected
location and format already provided via runtime context, skip both Q1 and Q2. The calling
workflow is responsible for asking both questions once; `implementation-planner` inherits the answers.

Specifically:
- If `plan_location` is set → skip Q1.
- If `plan_format` is set → skip Q2.
- If both are set → skip both; proceed directly to writing.

This ensures the user is never asked the same question twice in a single workflow.
